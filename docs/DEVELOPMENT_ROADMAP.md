## المحور الأول: نظام التدريس

---

### M1 — منهجية سقراط في التعليم

**المشكلة الحالية:** المعلم يشرح القاعدة مباشرةً قبل أن يتيح للطالب فرصة التفكير. هذا يُضعف الاستيعاب العميق ويُقلل من مشاركة الطالب.

**المخرج المطلوب:** المعلم يسأل الطالب أولاً، يستمع للإجابة، ثم يصحح ويشرح بناءً عليها — بدل الشرح المبتدئ.

**الأولوية:** 🔴 عالية | **التقدير:** أسبوع

#### المهام الفرعية

**M1.1 — تعديل system prompt لتبني أسلوب السؤال أولاً**
- إضافة تعليمات صريحة: "عند تقديم قاعدة جديدة، ابدأ بسؤال استكشافي"
- أمثلة في prompt: "كيف تقول X بالتركية؟" قبل شرح X
- تمييز بين الحالات: سؤال مباشر من الطالب (أجب) vs. موضوع جديد يُكتشف (اسأل)
- تجنب الأسلوب السقراطي حين الطالب محبط أو في عجلة (اقرأ النبرة)

**M1.2 — تعليمات انتظار الإجابة قبل الشرح**
- إذا سأل المعلم سؤالاً، **لا يعطي الإجابة في نفس الرسالة**
- ينتظر رد الطالب في الرسالة التالية ثم يبني عليها
- إضافة مؤشر في prompt: `[سؤال استكشافي — انتظر الرد]`

**M1.3 — تصنيف ردود الطالب وبناء الشرح عليها**
- إذا أجاب الطالب صحيحاً: "ممتاز! هذا بالضبط الصحيح. لنعمّق..."
- إذا أجاب خطأً جزئياً: "تقريباً صح! الجزء الأول كان صحيحاً، لكن..."
- إذا لم يعرف: "لا بأس، هذه قاعدة جديدة. لنتعلمها معاً..."
- إذا أجاب بالعربي بدل التركية: تشجيع المحاولة التركية

**M1.4 — قياس فعالية الأسلوب**
- مقارنة دقة الطالب في الأسابيع التي يُطبَّق فيها الأسلوب السقراطي
- تسجيل متى استخدم المعلم السؤال الاستكشافي في `api_logs`

**معايير القبول:**
- [ ] 60% من المواضيع الجديدة تبدأ بسؤال استكشافي
- [ ] المعلم لا يعطي إجابة كاملة في نفس رسالة السؤال
- [ ] الشرح يستشهد بإجابة الطالب ("كما قلت أنت...")

### M2 — خريطة المتطلبات السابقة (Prerequisites Map)

**المشكلة الحالية:** المعلم قد يشرح "حالة المفعول به" قبل أن يتأكد من إتقان "حالة الفاعل". لا توجد شجرة تبعيات للمهارات.

**المخرج المطلوب:** نظام يعرف أن مهارة B تتطلب إتقان مهارة A أولاً، ويوجه المعلم وفق ذلك.

**الأولوية:** 🟡 متوسطة | **التقدير:** أسبوعان

#### المهام الفرعية

**M2.1 — بناء شجرة مهارات اللغة التركية**
```
المستوى A1:
  ├── الحروف والنطق (لا يتطلب شيئاً)
  ├── التحيات (لا يتطلب شيئاً)
  ├── الأرقام (لا يتطلب شيئاً)
  ├── الفعل "olmak" في الحاضر (يتطلب: التحيات)
  └── حالة الفاعل/Nominative (يتطلب: الأفعال الأساسية)
      └── حالة المفعول/Accusative (يتطلب: Nominative)
          └── حالة الإضافة/Dative (يتطلب: Accusative)
              └── حالة الانتساب/Ablative (يتطلب: Dative)
```
- إنشاء ملف `lib/prerequisites.ts` يحتوي خريطة كاملة لـ A1→B2

**M2.2 — ربط weakness_analyzer بالخريطة**
- عند اكتشاف ضعف في مهارة B → التحقق أولاً من إتقان A
- إذا A ليست مُتقنة → توجيه المعلم لمعالجة A أولاً قبل B
- إضافة دالة `getBlockingPrerequisites(skill, masteredSkills[])`

**M2.3 — تحديث system prompt بقائمة المتطلبات المكتسبة**
- حقن قائمة "المهارات المُتقنة حتى الآن" في system prompt
- تحذير المعلم: "الطالب لم يُتقن بعد [X]، لا تتعمق في [Y]"

**M2.4 — واجهة عرض المسار في صفحة الأهداف**
- إضافة تبويب "مسار المهارات" في `/goals`
- شجرة بصرية تظهر ما أُتقن (أخضر) وما هو قادم (رمادي) وما يحتاج مراجعة (أحمر)

**معايير القبول:**
- [ ] ملف prerequisites.ts يحتوي 30+ مهارة مرتبة
- [ ] المعلم لا يشرح Dative قبل تأكيد إتقان Nominative
- [ ] صفحة الأهداف تعرض الشجرة البصرية

### M3 — التعلم الحلزوني (Spiral Learning)

**المشكلة الحالية:** موضوع يُدرَّس مرة واحدة ثم لا يعود إلا إذا أخطأ فيه الطالب. لا يوجد نظام لإعادة الزيارة بتعقيد متزايد.

**المخرج المطلوب:** كل موضوع يعود كل 2-3 أسابيع بمستوى أعمق وأكثر تركيباً.

**الأولوية:** 🟡 متوسطة | **التقدير:** أسبوعان

#### المهام الفرعية

**M3.1 — جدول مواضيع الحلزون**
- جدول في DB: `spiral_schedule(topic, level, last_taught_at, next_review_at, complexity_level)`
- complexity_level: 1=مقدمة، 2=تعميق، 3=تركيب، 4=إتقان
- دالة تحسب `next_review_at` بناءً على آخر جلسة ومستوى التعقيد

**M3.2 — كشف المواضيع المُدرَّسة من session_summaries**
- قراءة `topics_covered` من الملخصات الأخيرة
- تحديد المواضيع التي مضى عليها أكثر من أسبوعين
- تحديد مستوى التعقيد التالي المناسب

**M3.3 — تعليم المعلم بإعادة زيارة الموضوع**
- حقن في system prompt: "يوصى بمراجعة موضوع [X] هذه الجلسة بمستوى أعلى"
- تعليمات للمعلم: "الطالب تعلم [X] مبدئياً قبل 15 يوماً، الآن طبّقه في جمل مركبة"

**M3.4 — قياس التحسن عبر الجولات**
- مقارنة دقة الطالب في نفس الموضوع عبر جولاته المختلفة
- عرضها في Dashboard: "تحسنك في الأفعال الماضية: جولة 1: 45% → جولة 2: 70% → جولة 3: 88%"

**معايير القبول:**
- [ ] كل موضوع يُجدوَل للعودة تلقائياً بعد أسبوعين
- [ ] المعلم يذكر أن الموضوع قديم ويعمّقه بدل إعادة شرحه من الصفر
- [ ] Dashboard يعرض منحنى التحسن عبر الجولات

### M4 — كشف التعثر وتغيير الأسلوب

**المشكلة الحالية:** إذا استمر نفس الخطأ لأسابيع، المعلم يكرر نفس الشرح. لا يوجد تغيير في الاستراتيجية.

**المخرج المطلوب:** بعد 3 جلسات بنفس الخطأ، يغير المعلم أسلوبه تلقائياً (جدول، قصة، تمرين مختلف).

**الأولوية:** 🔴 عالية | **التقدير:** أسبوع

#### المهام الفرعية

**M4.1 — كشف التعثر في weakness_analyzer**
- إضافة تحليل زمني: هل الخطأ موجود في آخر 3 جلسات منفصلة؟
- إنشاء `StuckError { grammar_point, sessions_count, weeks_stuck, last_approach_used }`
- تمييز: خطأ عابر (مرة-مرتين) vs. خطأ متجذر (3+ جلسات)

**M4.2 — مكتبة أساليب تعليمية متعددة**
في `lib/teaching-strategies.ts`:
```typescript
type TeachingStrategy = 'explanation' | 'table' | 'story' | 'drill' | 'analogy' | 'examples_only'

const strategyRotation = {
  explanation: 'table',   // إذا فشل الشرح → جرّب الجدول
  table: 'story',          // إذا فشل الجدول → جرّب القصة
  story: 'drill',          // إذا فشلت القصة → جرّب التمرين المتكرر
  drill: 'analogy',        // إذا فشل التمرين → جرّب التشبيه
}
```

**M4.3 — تسجيل الأسلوب المستخدم**
- إضافة حقل `teaching_approach` في جدول `sessions` أو `session_summaries`
- تتبع: "آخر مرة شُرح فيها [topic] كان بأسلوب [table]"

**M4.4 — حقن التعليمات في system prompt**
- إذا الخطأ متجذر: "⚠️ الطالب عالق في [X] منذ 3 جلسات. الأسلوب الأخير كان [Y]. جرّب أسلوب [Z] هذه المرة"
- تعليمات محددة لكل أسلوب:
  - جدول: "أعطِ جدولاً مقارناً"
  - قصة: "ابنِ قصة قصيرة تستخدم القاعدة بشكل طبيعي"
  - تشبيه: "قارن بالعربية أو بشيء مألوف"

**M4.5 — تنبيه Telegram عند التعثر الطويل**
- إذا تعثر > 4 جلسات: رسالة للطالب عبر Telegram
  > "لاحظت أنك تواجه صعوبة في [X] منذ فترة. هل تريد جلسة مخصصة لهذا الموضوع؟"

**معايير القبول:**
- [ ] weakness_analyzer يميّز بين خطأ عابر ومتجذر
- [ ] system prompt يتضمن تعليمة تغيير الأسلوب بعد 3 جلسات
- [ ] تنبيه Telegram يُرسَل تلقائياً عند التعثر > 4 جلسات

### M5 — الملف الشخصي للطالب (Adaptive Student Model)

**المشكلة الحالية:** كل الطلاب يحصلون على نفس أسلوب التعليم. لا يوجد استنتاج لنمط تعلم الطالب أو تفضيلاته.

**المخرج المطلوب:** المعلم يستنتج تلقائياً كيف يتعلم هذا الطالب تحديداً ويتكيف معه.

**الأولوية:** 🟡 متوسطة | **التقدير:** أسبوعان

#### المهام الفرعية

**M5.1 — مخطط نمط التعلم**
إنشاء جدول `student_learning_profile`:
```sql
learning_style TEXT,         -- 'visual' | 'auditory' | 'kinesthetic' | 'reading'
optimal_session_length INT,  -- بالدقائق
preferred_time TEXT,         -- 'morning' | 'evening' | 'night'
topic_affinity TEXT[],       -- الموضوعات المفضلة
challenge_preference TEXT,   -- 'low' | 'medium' | 'high'
explanation_preference TEXT, -- 'brief' | 'detailed' | 'example-first'
last_updated_at TIMESTAMPTZ
```

**M5.2 — استنتاج النمط من البيانات الموجودة**
دالة `inferLearningProfile()` تعمل أسبوعياً:
- **بصري**: هل الطالب يتحسن أكثر بعد جداول؟
- **حركي**: هل دقته أعلى في جلسات التمارين vs. الشرح؟
- **الوقت الأمثل**: فحص timestamps للجلسات ذات الدقة الأعلى
- **طول الجلسة**: هل الدقة تنخفض بعد X رسالة؟

**M5.3 — حقن الملف الشخصي في system prompt**
```
نمط تعلم الطالب (مُستنتج من البيانات):
- يتعلم أفضل عبر: الأمثلة قبل القواعد
- الجلسة المثالية: ~25 دقيقة
- تفضيل الوقت: مسائي
→ قصّر شروحاتك واستخدم أمثلة تطبيقية قبل القاعدة
```

**M5.4 — إعدادات يدوية للطالب في صفحة الإعدادات**
إضافة قسم "كيف تتعلم؟" يسمح للطالب بتحديد:
- أفضّل: شرح القاعدة أولاً / الأمثلة أولاً / الاكتشاف
- جلساتي المفضلة: قصيرة مكثفة / طويلة هادئة
- تحديني: بشكل متكرر / أحياناً / نادراً

**معايير القبول:**
- [ ] الملف الشخصي يُحدَّث تلقائياً أسبوعياً
- [ ] system prompt يتضمن ملخص نمط التعلم
- [ ] الطالب يستطيع تعديل التفضيلات يدوياً

### M6 — تقييم CEFR تلقائي واقتراح الترقي

**المشكلة الحالية:** مستوى CEFR يتغير يدوياً فقط. المعلم لا يقترح الترقي تلقائياً حين يلاحظ التحسن.

**المخرج المطلوب:** نظام يحلل الأداء دورياً ويقترح تغيير المستوى مع دليل من البيانات.

**الأولوية:** 🟡 متوسطة | **التقدير:** أسبوع

#### المهام الفرعية

**M6.1 — معايير الترقي لكل مستوى**
في `lib/cefr-criteria.ts`:
```typescript
const promotionCriteria = {
  A1_to_A2: {
    minAccuracy: 80,          // دقة 80%+ في آخر 10 جلسات
    minSessions: 10,          // 10 جلسات على الأقل
    minVocabMastered: 150,    // 150 كلمة مُتقنة
    maxWeakGrammarPoints: 2,  // لا أكثر من نقطتين ضعف
    requiredSkills: ['nominative', 'basic_verbs', 'numbers']
  },
  // ... بقية المستويات
}
```

**M6.2 — دالة تقييم استعداد الترقي**
دالة `checkCEFRReadiness(currentLevel)` تعمل بعد كل 5 جلسات:
- تجمع: متوسط الدقة، عدد الجلسات، المفردات المُتقنة، نقاط الضعف
- تُقارن بمعايير الترقي
- تُعيد: `{ ready: boolean, score: number, missingCriteria: string[] }`

**M6.3 — اقتراح الترقي عبر Telegram**
عند تحقق المعايير:
```
🎉 تهانينا! بناءً على تحليل آخر 15 جلسة:
- دقتك: 83% ✅
- مفردات مُتقنة: 178 كلمة ✅
- نقاط ضعف: 1 فقط ✅

أرى أنك جاهز للانتقال من A1 إلى A2!
للتأكيد: أرسل "ترقّيني" / للاستمرار في A1: "ليس بعد"
```

**M6.4 — اختبار تحديد المستوى الأولي**
- عند أول استخدام: 10 أسئلة تكيفية تحدد المستوى الأولي
- بدل البدء من A1 افتراضياً
- أسئلة متدرجة (A1 → B2) تتوقف حين يفشل الطالب 3 متتاليات

**معايير القبول:**
- [ ] معايير موثقة لكل مستوى في cefr-criteria.ts
- [ ] اقتراح الترقي يُرسَل عبر Telegram تلقائياً
- [ ] اختبار التحديد الأولي يعمل في أول استخدام

### M7 — كشف الحالة العاطفية والتكيف

**المشكلة الحالية:** المعلم لا يميز بين طالب متحمس، محبط، متعجل، أو متعب.

**المخرج المطلوب:** المعلم يعدّل أسلوبه بناءً على الحالة المستنتجة.

**الأولوية:** 🟢 منخفضة | **التقدير:** 3 أيام

#### المهام الفرعية

**M7.1 — مؤشرات الحالة في system prompt**
تعليمات المعلم لاستنتاج الحالة من:
- طول الرسائل (قصيرة جداً = متعجل أو محبط)
- كثرة الأخطاء المتكررة في جلسة واحدة (= متعب)
- علامات الترقيم والمشاعر ("!")
- سؤال عن نفس الشيء مرتين = لم يفهم

**M7.2 — استجابات مخصصة لكل حالة**
```
متحمس → تحدّيه بمحتوى أصعب + "ممتاز! هل نجرب شيئاً أكثر صعوبة؟"
محبط → بسّط + شجّع + "هذا طبيعي، التركية صعبة في البداية، دعنا نحاولها بطريقة مختلفة"
متعجل → اختصر الشرح + ركّز على النتيجة فقط
متعب → "لاحظت أن ردودك أقل من المعتاد. هل نكتفي بمراجعة سريعة اليوم؟"
```

**M7.3 — تسجيل الحالة في session_summaries**
إضافة حقل `emotional_tone TEXT` في `session_summaries`
- يُستخرج تلقائياً ضمن ملخص الجلسة
- يُستخدم في التقرير الأسبوعي

**معايير القبول:**
- [ ] المعلم يغير نبرته حين يكتشف إشارات الإحباط
- [ ] لا يعطي تحديات إضافية حين يكتشف تعب الطالب

### M8 — بطاقات SRS بسياق المحادثة

**المشكلة الحالية:** بطاقات SRS تعرض "كلمة → ترجمة" بدون سياق. الطالب لا يتذكر من أين تعلم الكلمة.

**المخرج المطلوب:** كل بطاقة تعرض الجملة الأصلية التي تعلم فيها الكلمة من المحادثة.

**الأولوية:** 🔴 عالية | **التقدير:** 4 أيام

#### المهام الفرعية

**M8.1 — إضافة حقول السياق لجدول vocab_cards**
```sql
ALTER TABLE vocab_cards ADD COLUMN context_sentence TEXT;
ALTER TABLE vocab_cards ADD COLUMN context_session_id UUID REFERENCES sessions(id);
ALTER TABLE vocab_cards ADD COLUMN context_date TIMESTAMPTZ;
ALTER TABLE vocab_cards ADD COLUMN grammar_note TEXT;
```

**M8.2 — حفظ السياق عند إنشاء البطاقة**
في `app/api/feedback/route.ts`:
- عند استخراج كلمة جديدة من المحادثة، احفظ الجملة الكاملة كـ `context_sentence`
- احفظ `session_id` الحالية
- إضافة ملاحظة القاعدة النحوية إذا وُجدت

**M8.3 — تحديث واجهة SRS لعرض السياق**
في `app/review/page.tsx`:
- عرض بطاقة منقحة:
  ```
  ┌─────────────────────────────┐
  │  pazarlık                   │
  │  ─────────────────────────  │
  │  "Orada pazarlık yaptım"    │  ← الجملة الأصلية
  │  "تفاوضت على السعر هناك"   │
  │                             │
  │  📅 تعلمتها: 15 مارس       │
  │  📌 الدرس: المفاوضة        │
  └─────────────────────────────┘
  ```
- زر "عرض المحادثة" يأخذ إلى الجلسة الأصلية

**M8.4 — مزامنة SRS مع المحادثة**
- عند بدء جلسة محادثة: فحص البطاقات المستحقة اليوم
- إذا > 5 بطاقات مستحقة: تلميح للمعلم بنسج هذه الكلمات في المحادثة
- `[SRS_DUE: pazarlık, gitmek, yemek]` في context المعلم

**معايير القبول:**
- [ ] كل بطاقة جديدة تحفظ الجملة الأصلية
- [ ] واجهة SRS تعرض السياق والتاريخ
- [ ] المعلم يستخدم البطاقات المستحقة في المحادثة

### M9 — تمارين متعددة الأنواع
